## סשן 3 — הרצת LR ראשונה
**שלב:** שלב מקדים

הרצתי LR פעם ראשונה — קיבלתי AUC=0.97. הופתעתי. חשבתי שיש leakage. עברתי שלוש פעמים על ה-pipeline:

1. אין משתנה מטרה ב-X.
2. ה-train_test_split מסודר עם stratified ו-random_state.
3. StandardScaler מחושב על train בלבד דרך ה-Pipeline.
4. OneHotEncoder עם handle_unknown='ignore'.

לא מצאתי leakage. הסבר: ה-dataset הוא בעצם "תרגום" של נוסחה כלכלית פשוטה (`score = 38 + 0.6×margin% + 0.15×approval%`), ו-LR מצליח ללמוד את הנוסחה. בעיניים שלי, זה תקין כי זה משקף שה-feature engineering אכן ממקסם את הסיגנל.

## סשן 10 — Sensitivity Analysis באפליקציה
**שלב:** שלב הסְפְּרִינְט

בניתי טבלת ניתוח רגישות ל-Streamlit app. **חמישה תרחישים** נבחרו לבסוף:

```python
deltas = {
    "מחיר מכירה +10%":  {"מחיר_מכירה_למ2_חדש_ש\"ח": price_new * 1.10},
    "מחיר מכירה -10%":  {"מחיר_מכירה_למ2_חדש_ש\"ח": price_new * 0.90},
    "עלות בניה +15%":   {"מרווח_גולמי_%": _sens_margin(1.15),
                          'עלות_בנייה_ש"ח': construction_cost * 1.15},
    "עלות בניה -15%":   {"מרווח_גולמי_%": _sens_margin(0.85),

                          "משך_כולל_פרויקט_חודשים": total_months + 6},
```

הפונקציה `_sens_margin(cost_factor)` מחשבת מחדש את המרווח הגולמי כשעלות הבנייה משתנה — הכרחי כדי שהמודל יקבל קלט עקבי.

(הערה: ב-`calculator.html` השארתי את אותם חמישה תרחישים והוספתי תרחיש שישי — `+20% תוספת דירות`. ראה סשן 14.)

## סשן 13 — Refactoring ושמות
**שלב:** שלב הסְפְּרִינְט

הסרתי כל הערה שאינה קריטית. ב-`data/regenerate_dataset.py` שמרתי את משקלי נוסחת הניקוד ככתיבה ישירה (בלי להוציא לקבועים בעלי שם), כי הם נקראים פעם אחת בלבד והם זהים לנוסחה הרשמית של הפרויקט שמופיעה גם ב-README וב-`calculator.html`:

```python
margin_capped = df['מרווח_גולמי_%'].clip(-50, 80)
approval = df['הסתברות_אישור_%']
score_base = 38 + 0.6 * margin_capped + 0.15 * approval
score_noisy = score_base + np.random.normal(0, 4.0, len(df))
df['ציון_כדאיות_0_100'] = score_noisy.clip(0, 100).round(1)
```

המשקלים `38`, `0.6`, `0.15` הם "המסמך" — הם הנוסחה עצמה. במקום להוציא אותם לקבועים בעלי שם בקובץ הזה, וידאתי שהמספרים זהים בכל שלושת המקומות שבהם הם מופיעים (`regenerate_dataset.py`, `calculator.html`, `README.md`).

## סשן 14 — בניית calculator.html
**שלב:** שלב הסְפְּרִינְט (לקראת סופו)

Streamlit Cloud היה לא יציב בימים שלפני. החלטתי לבנות גיבוי שתמיד יעבוד — ובלי תלות בשרת בכלל.

החלטה חשובה שלקחתי: **לא לחלץ את מקדמי ה-LR ולשחזר sigmoid ב-JS**, אלא להשתמש בנוסחת הניקוד המקורית של ה-dataset הסינתטי:

```javascript
// מהקובץ regenerate_dataset.py
const marginCapped = Math.max(-50, Math.min(80, grossMargin));
const baseFromMargin = 0.6 * marginCapped;
const baseFromApproval = 0.15 * approval;
const scoreRaw = 38 + baseFromMargin + baseFromApproval;
const score = Math.max(0, Math.min(100, scoreRaw));
```

למה זה היה ההגיוני? כי ראיתי שה-LR למעשה לומד את הנוסחה הזו (זה מה שגרם ל-AUC הגבוה בסשן 3). אז במקום ליצור עוד שכבה של תרגום (LR coefs → JS sigmoid → אותה תוצאה בקירוב), פשוט מימשתי את הנוסחה ישירות. הוויתור היחיד הוא שאין SHAP אמיתי במחשבון — אבל יש לי "factor breakdown" משלי שמראה את התרומה היחסית של כל גורם בנקודות, וזה מספיק לקהל היעד של המחשבון העצמאי.

לוגיקה משלימה שהוספתי במחשבון:

```javascript
const CITY_APPROVAL = {
  "תל אביב-יפו": 82, "רמת גן": 80, ..., "קריית אתא": 54
};
const CITY_PRICE_FACTOR = { /* מקדמי מחיר יחסיים */ };
const TYPE_APPROVAL_DELTA = {
  "תמא 38/1": +8,   // פשוט יותר → יותר סיכוי לאישור
  "תמא 38/2": 0,
  "פינוי-בינוי": -4 // המורכב ביותר
};
```

תרחישי הרגישות במחשבון: **שמרתי את חמשת התרחישים שכבר היו לי באפליקציה והוספתי אחד שישי**. הסט המלא:

```javascript
const scenarios = [
  { name: "מחיר מכירה +10%",  mod: { priceNew: × 1.10 } },
  { name: "מחיר מכירה -10%",  mod: { priceNew: × 0.90 } },
  { name: "עלות בנייה +15%",  mod: { costPerSqm: × 1.15 } },
  { name: "עלות בנייה -15%",  mod: { costPerSqm: × 0.85 } },
  { name: "עיכוב 6 חודשים",   mod: { constructMonths: +6 } },
  { name: "+20% תוספת דירות", mod: { newUnits: × 1.20 } },
];
```

חמשת הראשונים זהים לסט שב-Streamlit (סשן 10), והוספתי את "+20% תוספת דירות" כי במחשבון העצמאי קל יותר לדמות שינוי בכמות הדירות החדשות (אין שם מודל ML שצריך לקבל קלט עקבי — רק נוסחה ישירה).

עיצוב: ערכת צבעים כהה (`--bg: #0f1923`), פונט Heebo, RTL מלא, תאימות mobile (`@media (max-width: 980px)`). שדות מספריים LTR, dropdowns RTL — אותו עיקרון מסשן 8.

תוצאות המחשבון תואמות את ה-Streamlit app בדיוק רב — שניהם מבוססים על אותה נוסחה, בסופו של דבר. הפער בין השניים הוא בעיקר בכך שב-Streamlit יש SHAP אמיתי על המודל המאומן, ובמחשבון יש פירוק גורמים שמחושב ישירות מהנוסחה.

## סשן 16 — ערכים קשיחים ב-app/main.py וההשלכות שלהם
**שלב:** תיעוד רטרואקטיבי (שלב הסְפְּרִינְט)

תיעוד מאוחר של החלטה שיווקית/מעצבית שלא תועדה בזמן אמת אבל משפיעה על תוצאות הכלי. ב-`app/main.py` (שורות 197-201) ארבעה ערכים אינם מוצגים למשתמש בטופס הקלט אלא מוזנים אוטומטית ברקע לפני הקריאה למודל:

```python
building_age  = 2024 - year_built
sales_months  = 12
price_old     = price_new * 0.82
irr           = 12.0
approval_pct  = 72.0
```

**הסיבות:**

- **`approval_pct = 72.0`** — הסתברות אישור התב״ע. בנוסחת הניקוד המשקל שלו הוא 0.15 (מתוך 100). העדפתי לקבע ערך ממוצע ולא להעמיס על המשתמש שדה נוסף שדורש הבנה רגולטורית.
- **`irr = 12.0`** — IRR משוער. הזנתי ערך אמצע סביר; ה-IRR האמיתי נגזר מהמרווח הגולמי ומשך הפרויקט, אבל בקלט המשתמש לא ביקשתי אותו כדי לפשט.
- **`price_old = price_new × 0.82`** — היחס המקובל בענף בין מחיר דירה ישנה לחדשה. במקום לבקש מהמשתמש שני מחירים, אני מחשב את הישן מהחדש.
- **`sales_months = 12`** — חודשי מכירות ממוצעים. ערך אופייני בענף.

**ההשלכה:** מאחר ש-`approval_pct` נכנס כקבוע 72.0, חלק מהשונות שהמודל אומן עליה לא בא לידי ביטוי באפליקציה — בפועל, רק `margin%` הוא ה-driver הדינמי של הציון. למי שירצה להפעיל את הכלי במצב "מומחה" עם שליטה מלאה — `calculator.html` חושף את `הסתברות אישור` כשדה קלט נפרד דרך מקדמי `CITY_APPROVAL`, ושם המשתמש יכול לראות את ההשפעה של מיקום על הציון.
